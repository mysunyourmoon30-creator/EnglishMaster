# v0.3.0 Acceptance Criteria

## Email Delivery Module

Functional:

- Operators can configure an email provider without committing secrets.
- The system can send email through the configured provider.
- Email delivery attempts record status, timestamps, recipient, subject, and bounded error details.
- Admins can manually retry failed email delivery.

Security:

- Provider credentials are read from secrets or environment configuration only.
- Responses and logs do not expose provider credentials, passwords, tokens, full connection strings, or full cookies.
- Email retry actions require appropriate admin permission.

Performance:

- Email delivery history is paginated or bounded.
- Retry operations do not block unrelated admin workflows for long-running provider calls.

Documentation:

- Provider configuration is documented.
- Secret handling and rotation guidance are documented.
- Failure and retry behavior is documented.

Test:

- Unit or integration tests cover successful send, failed send, retry, authorization, and secret-safe error handling.

## Notification Delivery Module

Functional:

- Notification delivery settings can be managed by authorized admins.
- Notifications that require email can create or link delivery attempts.
- Admins can view delivery status and retry failed notification email.

Security:

- Delivery management requires permissions.
- Notification payloads do not expose sensitive auth fields or private learner data to unauthorized users.

Performance:

- Delivery status lists are bounded.
- Retry operations are guarded against accidental bulk overload.

Documentation:

- Notification delivery settings and operational behavior are documented.

Test:

- Tests cover delivery settings, delivery status, retry, and authorization.

## Certificate Module

Functional:

- Admins can define certificate templates.
- The system can generate a certificate for a completed course.
- Students can view their certificates.
- Admins can manage issued certificates.

Security:

- Students can access only their own certificates unless explicitly authorized.
- Admin certificate management requires permission.
- Certificate generation cannot be forged by client-side state alone.

Performance:

- Certificate lists are paginated or bounded.
- Generation avoids loading unnecessary entity graphs.

Documentation:

- Completion rules, certificate generation, and admin/student workflows are documented.

Test:

- Tests cover generation, student ownership, admin management, and unauthorized access denial.

## Certificate Verification Module

Functional:

- Issued certificates have verification codes.
- Public verification can confirm valid certificate metadata without exposing private learner details.
- Revoked or invalid certificates are handled safely.

Security:

- Verification endpoint returns only public-safe fields.
- Verification codes are hard to guess.
- Internal identifiers and private learner data are not exposed.

Performance:

- Verification lookup is indexed or otherwise bounded.

Documentation:

- Public verification behavior and privacy boundaries are documented.

Test:

- Tests cover valid, invalid, revoked, and privacy-safe verification responses.

## Analytics Improvement Module

Functional:

- Admin metrics are more useful for learning and content operations.
- Student learning trend summary is available to the authenticated student.
- Analytics are based on privacy-safe aggregation.

Security:

- Student analytics are scoped to the authenticated learner.
- Admin analytics require reporting permissions.
- Sensitive auth fields, raw secrets, and private metadata are excluded.

Performance:

- Metrics queries are bounded and avoid unbounded full-graph loads.
- Exports are capped or streamed safely if included.

Documentation:

- Metric definitions and known limitations are documented.

Test:

- Tests cover admin metrics, student scoping, privacy, and bounded query behavior.
