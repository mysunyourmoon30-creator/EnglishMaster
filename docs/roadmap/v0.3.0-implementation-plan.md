# v0.3.0 Implementation Plan

## Status

`Planning complete; implementation not started`

## Goal

Deliver a small, safe v0.3.0 release that adds real notification delivery, completion evidence, and stronger admin/student insights.

## Selected Scope

1. External Email Provider + Notification Delivery.
2. Certificate System / Learning Completion Evidence.
3. Advanced Analytics Foundation.

## Phase 1: Email Provider Foundation

Goal:

- Add provider-backed email delivery foundations behind existing abstractions.

Scope:

- Provider abstraction.
- SMTP/configurable provider.
- Delivery status.
- Failed delivery capture.
- Manual retry foundation.

Out of scope:

- Marketing campaigns.
- Complex template editor.
- Bulk newsletter delivery.

Dependencies:

- Existing notification/email foundation.
- Environment variable and secret configuration docs.

Risks:

- Secret leakage.
- Provider lock-in.
- Retry abuse.
- Long-running provider calls.

Acceptance criteria:

- Email can be sent through configured provider.
- Failed sends are recorded safely.
- Retry is permission-protected and bounded.
- Secrets are never returned to clients.

Test requirements:

- Fake sender integration tests.
- Success/failure/retry tests.
- Authorization tests.
- Secret masking tests.

Docs requirements:

- Provider configuration.
- Secret handling.
- Retry and failure behavior.

Rollback notes:

- Disable provider delivery while retaining delivery records.
- Keep existing notification records intact.

## Phase 2: Notification Delivery Integration

Goal:

- Connect notifications to email delivery where appropriate.

Scope:

- Notification delivery settings.
- Delivery status visibility.
- Retry failed notification email.

Out of scope:

- Push notifications.
- SMS.
- Campaign scheduling.

Dependencies:

- Phase 1 email delivery foundation.

Risks:

- Duplicate sends.
- Notification payload privacy.
- Retry overload.

Acceptance criteria:

- Authorized admins can manage delivery settings.
- Notification email delivery is tracked.
- Failed delivery can be retried safely.

Test requirements:

- Delivery setting tests.
- Notification-to-email integration tests.
- Authorization tests.

Docs requirements:

- Notification delivery operations and troubleshooting.

Rollback notes:

- Disable notification email delivery without deleting notifications.

## Phase 3: Certificate Foundation

Goal:

- Add completion evidence through issued certificates.

Scope:

- Certificate template foundation.
- Course completion certificate generation.
- Student certificate page.
- Admin certificate management page.

Out of scope:

- Paid certificates.
- Marketplace credentials.
- Advanced PDF designer.

Dependencies:

- Course/lesson/quiz completion rules.
- Student identity and ownership boundaries.

Risks:

- Forged completion.
- Duplicate certificates.
- Private data exposure.

Acceptance criteria:

- Certificates generate only from server-side completion rules.
- Students see only their own certificates.
- Admin management requires permission.
- Duplicate certificates are prevented.

Test requirements:

- Generation tests.
- Ownership tests.
- Admin permission tests.
- Duplicate prevention tests.

Docs requirements:

- Completion rules.
- Admin/student certificate workflows.

Rollback notes:

- Preserve issued certificate metadata.
- Avoid destructive rollback of completion evidence.

## Phase 4: Certificate Verification

Goal:

- Let public visitors verify certificates safely.

Scope:

- Verification code.
- Public verification endpoint/page.
- Safe certificate verification DTO.

Out of scope:

- Social sharing.
- Blockchain/third-party credential networks.

Dependencies:

- Phase 3 certificate issuance.

Risks:

- Guessable codes.
- Learner privacy exposure.
- Internal ID exposure.

Acceptance criteria:

- Verification codes are difficult to guess.
- Public verification returns only public-safe fields.
- Revoked/invalid certificates are handled safely.

Test requirements:

- Valid/invalid/revoked verification tests.
- Privacy-safe response tests.

Docs requirements:

- Public verification behavior and privacy boundaries.

Rollback notes:

- Verification can be disabled while preserving certificates.

## Phase 5: Advanced Analytics Foundation

Goal:

- Improve admin metrics and student trend summaries without overexposing data.

Scope:

- Admin metrics improvements.
- Student learning trend summary.
- Privacy-safe aggregation.
- Optional CSV export if scope allows.

Out of scope:

- Data warehouse.
- Real-time analytics engine.
- AI personalization rollout.

Dependencies:

- Existing activity, reporting, quiz, practice, study plan, and learning report data.

Risks:

- Expensive queries.
- Sensitive data exposure.
- Unbounded exports.

Acceptance criteria:

- Admin analytics require permission.
- Student trends are scoped to current student.
- Metrics are bounded and aggregate-oriented.
- Sensitive metadata is excluded.

Test requirements:

- Query/aggregation tests.
- Authorization tests.
- Student ownership tests.
- Export cap tests if CSV is included.

Docs requirements:

- Metric definitions and limitations.

Rollback notes:

- Analytics UI/endpoints can be disabled without affecting core learning data.

## Phase 6: v0.3.0 Stabilization

Goal:

- Stabilize, audit, document, and prepare v0.3.0 release candidate.

Scope:

- Final build/test pass.
- Security audit.
- Release docs.
- RC tag prep.

Out of scope:

- New feature expansion.
- Post-scope polish unless release-blocking.

Dependencies:

- Phases 1-5 completed and reviewed.

Risks:

- Scope creep.
- Missing permission tests.
- Missing staging validation.

Acceptance criteria:

- Build/test pass.
- Security audit complete.
- Docs complete.
- Known limitations updated.
- RC tag instructions prepared but not executed automatically.

Test requirements:

- Full test suite.
- Focused module tests.
- Manual staging smoke plan.

Docs requirements:

- Release notes.
- Known limitations.
- Smoke/UAT plan.
- Rollback/hotfix notes.

Rollback notes:

- Prepare module-specific rollback notes before RC.
