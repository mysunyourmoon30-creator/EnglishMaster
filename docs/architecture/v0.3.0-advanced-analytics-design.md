# v0.3.0 Advanced Analytics Design

## Purpose

Advanced analytics extends the existing reporting module from operational snapshots into learner and administrator insight surfaces. It stays inside the modular monolith and uses EnglishMaster application data only.

The v0.3.0 goal is useful product analytics, not a warehouse, machine learning platform, or external BI integration.

## Scope

Included:

- Learning analytics aggregation over existing learning, practice, quiz, goal, streak, achievement, certificate, and report data.
- Admin analytics dashboard with aggregate engagement, completion, assessment, and certificate metrics.
- Student analytics dashboard with authenticated learner-owned trends and recommendations.
- Privacy-safe DTOs and query limits.
- Read-only APIs and Blazor pages.

Not included:

- External analytics services.
- Event streaming infrastructure.
- Predictive models.
- Export/report builder.
- Cross-tenant analytics.
- Raw learner data dumps.

## Data Sources

Initial analytics should derive from:

- `StudentProfile`
- `LearningActivityLog`
- `LessonProgress`
- `CourseProgress`
- `BookProgress`
- `QuizAttempt`
- `PracticeSession`
- `PracticeItem`
- `LearningGoal`
- `DailyStudyPlan`
- `StudentStreak`
- `StudentAchievement`
- `WeeklyLearningReport`
- `IssuedCertificate`

The first implementation should use live read-only EF queries. Summary tables or cached rollups are allowed later only if measured latency requires them.

## Application Boundary

Create a dedicated analytics feature area:

```text
EnglishMaster.Application.Features.Analytics
EnglishMaster.Infrastructure.Analytics
EnglishMaster.Contracts.Analytics
EnglishMaster.Api.Endpoints.AnalyticsEndpoints
EnglishMaster.Web.Services.Analytics
EnglishMaster.Web.Components.Pages.Analytics
```

Repository interface:

```text
IAnalyticsRepository
```

Handlers:

```text
AnalyticsQueryHandler
```

The API endpoint must remain thin: bind route/query/user context, call handlers, map result DTOs.

## API Shape

Admin routes:

```text
GET /api/v1/admin/analytics/overview
GET /api/v1/admin/analytics/engagement
GET /api/v1/admin/analytics/completions
GET /api/v1/admin/analytics/assessments
GET /api/v1/admin/analytics/certificates
```

Student routes:

```text
GET /api/v1/me/analytics/overview
GET /api/v1/me/analytics/activity
GET /api/v1/me/analytics/progress
GET /api/v1/me/analytics/assessments
```

Recommended query parameters:

```text
fromDate
toDate
period=day|week|month
limit
```

Defaults:

- Default period: last 30 days.
- Maximum range: 366 days.
- Maximum series points: 60.
- Maximum item list size: 100.

## Permissions

Use existing `reports.read` for admin analytics in v0.3.0. Add a separate analytics permission only if admin roles need analytics without reporting access.

Student analytics uses authenticated user context and does not require an additional permission.

## Privacy Rules

Admin analytics:

- Prefer aggregate metrics.
- Avoid learner names and emails in dashboards.
- Do not expose raw quiz answers or answer correctness details.
- Keep recent activity summaries bounded and purpose-specific.

Student analytics:

- Use only current learner-owned data.
- Do not expose other learners through joins or counts that can identify individuals in tiny cohorts.
- Keep recommendation text deterministic and rule-based.

Public analytics:

- No public analytics endpoints in v0.3.0.

## Metrics

Admin overview:

- Active learners.
- Study minutes.
- Learning activities.
- Course completions.
- Quiz attempts.
- Average quiz score.
- Practice sessions completed.
- Certificates issued.

Engagement:

- Active learner trend.
- Study minutes trend.
- Activity count trend by content type.
- Study plan completion rate.

Completions:

- Lesson, course, and book starts/completions.
- Completion rate by content type.
- Top completed courses.

Assessments:

- Quiz attempt count.
- Pass rate.
- Average score.
- Low-performing quiz list by aggregate score.

Certificates:

- Issued count.
- Revoked count.
- Issue trend.
- Top certificate courses.

Student overview:

- Total study minutes.
- Current streak and longest streak.
- Completed lessons/courses/books.
- Quiz pass rate and average score.
- Practice completion.
- Certificates earned.

Student trends:

- Study minutes over time.
- Activity by content type.
- Course progress.
- Quiz scores over time.

## Performance Guardrails

- Use `AsNoTracking`.
- Use projection DTOs, not entity graph loading.
- Filter by date range before grouping.
- Keep grouping in database where EF can translate cleanly.
- Clamp date ranges and list sizes in application handlers.
- Add indexes only when a real query needs them; prefer existing indexes first.

## UI Direction

Admin analytics should be compact and dashboard-like:

- Metric strips.
- Simple trend tables or lightweight chart-ready DTOs.
- Dense comparison sections.
- No marketing-style hero sections.

Student analytics should feel progress-oriented:

- Personal summary cards.
- Recent trend sections.
- Clear empty states for new learners.
- No comparison to other learners in v0.3.0.

## Verification Strategy

- Integration tests for admin endpoint permission and aggregate correctness.
- Integration tests for student endpoint current-user scoping.
- Focused repository tests are optional unless query complexity grows.
