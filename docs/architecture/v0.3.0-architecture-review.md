# v0.3.0 Architecture Review

## Architecture Fit Result

`Go for implementation planning with guardrails`

The selected v0.3.0 modules fit the existing Clean Architecture direction if boundaries remain strict:

- Domain owns core concepts and invariants.
- Application owns use cases, validation, authorization orchestration, and DTO mapping.
- Infrastructure owns SMTP/provider implementations, storage integrations, EF Core persistence, and PDF/provider details.
- API exposes safe, permission-protected endpoints.
- Blazor uses API/Application abstractions and must not access `DbContext` directly.

## Module Fit

| Module | Fit | Notes |
| --- | --- | --- |
| Email Delivery Module | Good with provider abstraction | Keep provider-specific SDKs in Infrastructure. |
| Notification Delivery Module | Good | Build on existing notification/email foundations. |
| Certificate Module | Good with careful domain rules | Certificate issuing rules belong in Domain/Application; rendering/storage details belong in Infrastructure. |
| Certificate Verification Module | Good with public-safe DTOs | Public endpoint must expose only verification-safe fields. |
| Analytics Improvement Module | Good if bounded | Keep queries paginated, aggregated, and privacy-safe. |

## Email Risk Summary

- SMTP/provider configuration must come from environment variables or secrets.
- Provider credentials must never be returned to UI/API callers.
- Retry behavior needs idempotency and clear delivery attempt records.
- Failed email handling must store bounded, sanitized error details.
- Templates must avoid rendering raw user-controlled HTML unless sanitized.
- Provider-specific lock-in should be isolated behind Application abstractions and Infrastructure adapters.

## Certificate Risk Summary

- Certificate generation must be based on server-side completion rules.
- Verification codes must be non-sequential and difficult to guess.
- Public verification must not expose private learner data, internal IDs, or admin notes.
- Duplicate certificate prevention needs a unique course/student/completion boundary.
- PDF generation should be treated as an output adapter, not a Domain concern.
- Template data must be constrained and sanitized.

## Analytics Risk Summary

- Dashboard metrics must avoid expensive full-graph queries.
- Analytics endpoints should return aggregates and trends, not raw sensitive learner records.
- Lists and exports must be paginated, bounded, or capped.
- Query plans should be reviewed before release if metrics touch large tables.
- Student analytics must be scoped to the authenticated student profile.

## Implementation Guardrails

- No direct `DbContext` access from Blazor pages.
- No provider SDK references outside Infrastructure.
- No secrets in DTOs, logs, API responses, or UI state.
- No public certificate endpoint that exposes private learner fields.
- No analytics endpoint without explicit authorization and bounded query behavior.

## Go / No-Go Recommendation

`Go for implementation planning`.

Do not start coding until endpoint boundaries, permission names, data ownership rules, and test coverage expectations are documented in the implementation plan.
