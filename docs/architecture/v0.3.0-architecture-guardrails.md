# v0.3.0 Architecture Guardrails

## Status

`Approved for design prompts; feature implementation still requires explicit approval`

This document records Prompt 163 - v0.3.0 Architecture Guardrails.

## Baseline Architecture

EnglishMaster keeps the current Clean Architecture and modular monolith layout:

| Layer | Responsibility |
| --- | --- |
| Domain | Business entities, invariants, value rules, status transitions, and domain-only validation. |
| Application | Use cases, command/query handlers, repository/service abstractions, DTOs, orchestration, and transaction intent. |
| Infrastructure | EF Core repositories, provider adapters, storage, external integrations, configuration binding, and framework-specific implementations. |
| Api | Minimal API endpoints, auth/permission checks, request parsing, response mapping, and composition. |
| Contracts | Public request/response DTOs shared with clients. |
| Web | Blazor pages/components and API clients. No direct persistence access. |

## Dependency Rules

- Domain must not reference EF Core, ASP.NET Core, Blazor, HTTP, provider SDKs, configuration, or persistence concerns.
- Application may define interfaces for repositories, provider gateways, queue processors, analytics readers, and certificate services.
- Infrastructure implements Application interfaces and owns provider SDKs, EF Core, SQL Server, file/PDF/email integrations, and configuration binding.
- Api maps endpoints to Application handlers and must not contain business rules beyond request/auth/response handling.
- Web calls API clients and must not use `DbContext`, Infrastructure services, or Domain entities directly.
- Contracts must not expose EF entities, provider payloads, secrets, internal IDs for public verification, or private learner details.

## Module Boundaries

### Notification Delivery

Owned areas:

- Existing `Notifications` and `EmailMessages` foundations.
- Email provider abstraction in Application.
- Provider implementation in Infrastructure.
- Delivery queue and retry state as bounded notification/email delivery concepts.

Rules:

- Provider-specific code stays in Infrastructure.
- Delivery attempts must be durable and idempotency-aware.
- Retry operations must be bounded and permission-protected.
- Failure details must be sanitized and capped.
- Secrets are never stored in Domain entities, returned in DTOs, or logged.

### Certificate System

Owned areas:

- Certificate templates.
- Issued certificates.
- Certificate generation from server-side completion rules.
- Student-owned certificate list.
- Admin certificate management.
- Public certificate verification.

Rules:

- Certificate issuance rules belong in Domain/Application, not Web or Api endpoints.
- Rendering, PDF output, and file storage are Infrastructure adapters.
- Public verification uses high-entropy verification codes and public-safe DTOs.
- Internal IDs, private learner data, admin notes, and raw completion records must not be exposed publicly.
- Duplicate prevention must be enforced in Domain/Application and backed by persistence constraints.

### Advanced Analytics

Owned areas:

- Admin aggregate metrics.
- Student-owned learning trends.
- Bounded exports if included.

Rules:

- Analytics endpoints return aggregates or bounded summaries, not raw learner event streams.
- Admin analytics require explicit reporting/analytics permission.
- Student analytics are scoped to the authenticated student.
- Queries must be paginated, date-bounded, or capped.
- Large query shapes require review before release.

## Permissions

Add dedicated permissions only when existing permissions do not describe the action clearly.

Recommended permission shape:

| Area | Permission Intent |
| --- | --- |
| Email provider settings | Manage provider configuration status and test sends. |
| Notification delivery | View delivery status, process queue, and retry failed sends. |
| Certificate management | Manage templates and administrative certificate actions. |
| Certificate verification | Public verification should not require auth but must use public-safe DTOs. |
| Admin analytics | View admin aggregate analytics and exports. |
| Student analytics | Authenticated learner-owned access only; no admin permission needed for own data. |

Every new admin endpoint must have authorization tests. Every learner-owned endpoint must have cross-user privacy tests.

## Persistence Guardrails

- New durable concepts require EF Core configurations and migrations before release.
- Unique constraints are required for duplicate-sensitive records such as issued certificates and delivery idempotency keys.
- Date/time values must use UTC and `TimeProvider` where current time is needed.
- Queue state transitions must be explicit and testable.
- Retry counters, timestamps, and sanitized failure summaries must be bounded.
- Avoid storing full provider payloads unless they are sanitized and operationally required.

## Endpoint Guardrails

- Use route groups consistent with existing endpoint files.
- Keep public routes isolated from authenticated/admin routes.
- Public certificate verification must be read-only and rate-limit friendly.
- Mutating admin actions must require explicit permissions.
- Endpoint handlers should delegate to Application handlers and map results to HTTP responses.
- Invalid IDs, invalid dates, and invalid verification codes should return safe errors without revealing existence of private records.

## Web Guardrails

- Add API clients under `EnglishMaster.Web/Services/<Feature>`.
- Keep pages/components focused on UI state and user interaction.
- Do not put business rules, completion rules, queue processing, or provider logic in Blazor.
- UI must not display provider secrets or private verification data.
- Admin pages must handle forbidden/not-found states clearly.

## Testing Requirements

| Area | Required Tests |
| --- | --- |
| Email provider | Fake provider success/failure, secret masking, settings access, retry bounds. |
| Notification queue | Enqueue/process/retry/failure transitions, duplicate prevention, authorization. |
| Certificates | Generation rules, duplicate prevention, ownership, admin permissions. |
| Public verification | Valid/invalid/revoked codes, public-safe DTOs, no private data leakage. |
| Analytics | Authorization, student ownership, bounded queries, aggregate correctness. |
| Architecture | Add architecture tests if new project/layer references risk dependency violations. |

## Release Guardrails

- v0.3.0 implementation must not deploy production.
- v0.3.0 release tags must not be created before Prompt 185.
- v0.2.0 production readiness remains a separate gate.
- Any new secret, provider setting, migration, queue processor, or public endpoint needs release documentation before RC.

## No-Go Conditions

- Provider SDK referenced outside Infrastructure.
- Secrets exposed in DTOs, UI, logs, or tests.
- Public verification exposing private learner fields.
- Analytics returning unbounded learner-level data.
- Blazor accessing `DbContext` or Infrastructure repositories directly.
- Missing authorization tests for new admin endpoints.
- Missing ownership tests for learner-owned endpoints.
