# v0.3.0 Certificate System Design

## Status

`Designed; implementation pending`

This document records Prompt 170 - Certificate System Design.

## Goals

- Provide completion evidence for learners who complete courses.
- Let admins manage certificate templates.
- Let students view their own issued certificates.
- Let public visitors verify certificates safely.
- Prevent duplicate certificates for the same learner/course completion.

## Existing Completion Evidence

The first v0.3.0 certificate rule uses existing server-side course progress:

- `CourseProgress.UserId`
- `CourseProgress.CourseId`
- `CourseProgress.Status == Completed`
- `CourseProgress.ProgressPercent == 100`

Completion is evaluated server-side in Application/Infrastructure. UI state must not issue certificates directly.

## Domain Model

### CertificateTemplate

Purpose: admin-managed template metadata.

Fields:

- `Id`
- `Code`
- `Name`
- `Description`
- `BodyTemplate`
- `IsActive`
- `CreatedAt`
- `UpdatedAt`

Rules:

- Code and name are required.
- Body template is bounded.
- Only active templates can be used for new issuance.

### IssuedCertificate

Purpose: durable completion evidence.

Fields:

- `Id`
- `StudentUserId`
- `CourseId`
- `TemplateId`
- `CertificateNumber`
- `VerificationCode`
- `StudentDisplayName`
- `CourseTitle`
- `IssuedAt`
- `RevokedAt`
- `RevocationReason`
- `CreatedAt`
- `UpdatedAt`

Rules:

- Student user, course, template, certificate number, and verification code are required.
- Verification code must be high entropy and non-sequential.
- One active certificate per student/course.
- Public verification must not expose internal IDs, user IDs, or admin-only notes.
- Revoked certificates verify as revoked without exposing private details.

Expected folder:

```text
EnglishMaster.Infrastructure/Certificates
```

Responsibilities:

- EF repository.

- Verification code generation.
- EF configurations and migrations.

No PDF rendering is required in the first certificate slice unless explicitly added later.

## API Routes

| Route | Auth | Purpose |
| --- | --- | --- |
| `GET /api/v1/admin/certificate-templates` | Certificate manage/read permission | Search templates. |
| `POST /api/v1/admin/certificate-templates` | Certificate manage permission | Create template. |
| `PUT /api/v1/admin/certificate-templates/{id}` | Certificate manage permission | Update template. |
| `POST /api/v1/me/certificates/course/{courseId}/generate` | Authenticated learner | Generate own course certificate if eligible. |
| `GET /api/v1/me/certificates` | Authenticated learner | List own certificates. |
| `GET /api/v1/admin/certificates` | Certificate manage/read permission | Admin certificate search. |
| `GET /api/v1/certificates/verify/{code}` | Public | Public-safe verification. |

## Contracts

Use separate DTOs:

- Admin template DTO.
- Student certificate DTO.
- Admin certificate DTO.
- Public verification DTO.

Public verification DTO must include only:

- Certificate number.
- Public verification status.
- Student display name, if accepted by product.
- Course title.
- Issued date.
- Revoked status.

Do not expose:

- Student user ID.
- Course internal ID.
- Template internal ID.
- Admin notes.
- Raw completion records.

## Permissions

Add certificate permissions:

- `certificates.read`
- `certificates.manage`

Student-owned certificate list/generation requires authentication and ownership checks, not admin permissions.

## Persistence Constraints

Required indexes:

- Unique `CertificateTemplate.Code`.
- Unique `IssuedCertificate.CertificateNumber`.
- Unique `IssuedCertificate.VerificationCode`.
- Unique active certificate boundary for `StudentUserId + CourseId` where feasible.

If filtered unique indexes are not implemented in the first migration, Application must enforce one active certificate per student/course and the limitation must be documented for release review.

## Tests

Required implementation tests:

- Template create/update validation.
- Only active template can issue.
- Completed course can generate certificate.
- Incomplete course cannot generate certificate.
- Duplicate generation returns existing certificate or validation result, not a duplicate.
- Student sees only own certificates.
- Admin endpoints require certificate permission.
- Public verification returns safe fields only.
- Invalid verification code returns safe not-found response.

## Rollback

- Certificate tables can remain dormant if routes are disabled.
- Do not delete issued certificate records during rollback unless directed by an incident runbook.
- Public verification can be disabled separately from admin/student certificate records.
