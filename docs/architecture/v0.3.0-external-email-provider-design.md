# v0.3.0 External Email Provider Design

## Status

`Designed; implementation pending`

This document records Prompt 164 - External Email Provider Design.

## Existing Foundation

EnglishMaster already has:

- `EmailMessage` domain entity with Pending, Sent, and Failed states.
- `IEmailMessageRepository` for queue/search/status updates.
- `IEmailSender` Application abstraction.
- `DevelopmentEmailSender` Infrastructure adapter.
- Admin email endpoints for search and queue.
- Email read/manage permissions.

v0.3.0 should extend this foundation instead of replacing it.

## Goals

- Support a configurable external email provider.
- Keep provider-specific implementation in Infrastructure.
- Never expose provider secrets through API, UI, logs, DTOs, or tests.
- Provide configuration status and test-send capability.
- Preserve the current development sender as the local fallback.

## Provider Model

Use a provider selector in Infrastructure:

| Provider | Purpose |
| --- | --- |
| `Development` | Local/dev logging sender. Default when no provider is configured. |
| `Smtp` | External SMTP delivery using configuration and secrets. |

Future providers can be added behind the same Application abstraction.

## Application Contracts

Extend the Application layer with narrow contracts:

```csharp
public sealed record EmailSendRequest(
    string ToEmail,
    string? ToName,
    string Subject,
    string Body,
    bool IsHtml);

public sealed record EmailProviderStatusDto(
    string Provider,
    bool IsConfigured,
    string FromEmail,
    string FromName,
    bool SupportsTestSend);

public interface IEmailSender
{
    Task SendAsync(EmailSendRequest request, CancellationToken cancellationToken);
}

public interface IEmailProviderStatusService
{
    EmailProviderStatusDto GetStatus();
}
```

`EmailMessageDto` may still be used by command/query responses, but provider sending should use a narrower `EmailSendRequest` so retry and queue processing can send without exposing persistence-shaped DTOs to provider adapters.

## Configuration Shape

Recommended configuration:

```json
{
  "Email": {
    "Provider": "Development",
    "FromEmail": "noreply@example.com",
    "FromName": "EnglishMaster",
    "Smtp": {
      "Host": "",
      "Port": 587,
      "UseSsl": true,
      "UserName": "",
      "Password": ""
    }
  }
}
```

Rules:

- `Password` must come from environment variables or deployment secrets outside production examples.
- Status responses may include provider, configured flag, from email, from name, and capability flags.
- Status responses must not include SMTP host, username, password, connection strings, tokens, or raw provider diagnostics.

## Command Flow

### Queue Email

Existing queue behavior stays:

1. Admin calls queue endpoint.
2. Application validates and creates `EmailMessage`.
3. Repository persists Pending email.

### Test Send

New test-send behavior:

1. Admin with email manage permission submits recipient/subject/body.
2. Application builds `EmailSendRequest`.
3. Infrastructure provider sends the message.
4. API returns success or sanitized validation/problem response.

Test send should not persist a normal queued email unless a later prompt decides to add audit records.

### Dispatch Pending Email

Prompt 166 will add durable queue processing. Prompt 165 should prepare provider sending only.

## Infrastructure Design

Add:

- `EmailOptions`
- `SmtpEmailOptions`
- `EmailProviderStatusService`
- `SmtpEmailSender`
- `ConfiguredEmailSender` or registration logic that chooses Development vs SMTP.

Use framework-provided SMTP support if available. Do not add a provider package unless implementation cannot be completed with the existing .NET stack.

## Security Rules

- Never log secrets.
- Log recipient and message ID only at normal information level.
- Store sanitized failure summaries only.
- Do not include provider exception details in API responses.
- Validate email addresses using conservative basic checks; do not attempt full RFC parsing in v0.3.0.
- Test-send endpoint must require `Permissions.EmailManage`.

## Tests Required In Prompt 165

- Provider status returns masked/non-secret status.
- Development provider is selected by default.
- SMTP provider reports unconfigured when required settings are missing.
- Test-send endpoint requires email manage permission.
- Test-send validates required fields.
- Provider failure returns sanitized result.

## Rollback

- Set `Email:Provider` to `Development` to disable external delivery.
- Existing queued email records remain intact.
- No schema change is required for provider selection in Prompt 165.
