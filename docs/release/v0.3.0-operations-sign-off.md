## Purpose

`docs/release/v0.3.0-final-tag-prep.md` lists "Operations sign-off" as a release gate but no document for it has ever existed — for this release or the prior one (v0.2.0's own go-live checklist left the same line unchecked with no follow-up). ## Reconciliation: Operational Risk Doc vs. What Was Actually Built

`docs/operations/v0.3.0-operational-risk.md` was written earlier in the v0.3.0 cycle, before implementation. Two harmless drifts between that plan and what was actually built, worth recording rather than leaving silently inconsistent:

1. It anticipated a **"public certificate verification base URL"** configuration key. None exists, and none is needed — verification is exposed as a relative route (`GET /api/v1/public/certificates/{verificationCode}`, and the Web page `/certificates/verify/{code}`) resolved against whatever host serves the app. Confirmed during the go-live checklist review this session.
2. It anticipated **"delivery retry limits"** (implying a numeric retry-count/backoff policy). What was actually built is simpler: `EmailMessageCommandHandler.RetryFailedAsync` is a single explicit, permission-protected manual retry per message — no automatic backoff or attempt counter. This still satisfies the risk doc's own stated requirement ("explicit, permission-protected, and bounded" — one manual attempt is bounded), just not via a numeric limit setting.

Neither is a defect. Both are normal plan-vs-built drift that a pre-implementation risk document should expect.

| Production backup automation / scheduling | Not configured | `docs/testing/v0.3.0-backup-restore-verification.md` proved the backup/restore *mechanism* works, but does not configure or schedule anything for a real production host — that's host/platform-specific operational setup. |
| Real SMTP provider account, credentials, sending-domain approval (SPF/DKIM, reputation) | Not provisioned | `docs/release/v0.3.0-production-go-live-checklist.md` — the adapter (MailKit-based `SmtpEmailSender`) and the automatic `EmailDeliveryWorker` are both code-complete and tested; nobody has set up a live provider account yet. |
| File storage backup with real media files | Not exercised | No real media files existed in this cycle's test data (fixtures were seeded via API/SQL, not file upload) — the documented procedure exists but hasn't been run against a non-trivial media library. |

## Decision

Do not treat "Operations sign-off" as complete based on this document alone. What this document establishes: the operational documentation (runbook, DR playbooks, monitoring doc, backup procedures) is **current, internally consistent, and references real working parts of v0.3.0** — an operator picking these up today would not be following stale or incorrect instructions. What it does not establish, and cannot: that an actual operations owner has read, accepted, and staffed for this responsibility, or that the explicitly-open items above have been resolved. Formal sign-off — a named person accepting operational responsibility for v0.3.0 in production — remains a human action, structurally the same as release-owner approval and rollback-owner assignment in `docs/release/v0.3.0-production-go-live-checklist.md`.
