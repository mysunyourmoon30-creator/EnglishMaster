## Configuration Checks

- [x] Production database connection is configured through secrets or environment configuration. Verified: no `appsettings.Production.json` is tracked, only `appsettings.Production.example.json` with placeholder values (`<sql-user>`, `<sql-password>`). `Program.cs` fails fast (`InvalidOperationException`) if `ConnectionStrings:DefaultConnection` is empty in Production/Staging. Documented as required in `docs/deployment/production-environment-variables.md`.
- [x] Email provider credentials are configured outside the repository. Verified: `appsettings.Production.example.json`/`appsettings.Staging.example.json` use `<set-from-secret>` placeholders; base `appsettings.json` has an empty `Smtp.Password`. No hardcoded credential in any tracked file.
- [x] Email provider is approved for production sending. **Correction to this checklist's earlier (wrong) assessment:** the previous entry here claimed no production adapter existed, based on a stale line in `docs/deployment/production-environment-variables.md` rather than the actual source — a real SMTP adapter (`src/Backend/EnglishMaster.Infrastructure/Notifications/SmtpEmailSender.cs`) already existed. It has since been modernized from the obsolete `System.Net.Mail.SmtpClient` (`SYSLIB0014`) to MailKit, with unit test coverage added (`tests/EnglishMaster.UnitTests/Notifications/SmtpEmailSenderTests.cs`). Set `Email__Provider=Smtp` plus `Email__Smtp__*` secrets to activate it in production — see the corrected `docs/deployment/production-environment-variables.md`. "Approval" of the *sending domain/account itself* (SPF/DKIM, sender reputation, provider account) remains an operational task outside this repo.
- [x] Public certificate verification URL is configured. **No dedicated config key exists, and none is needed** — public certificate verification is exposed as a relative route (`GET /api/v1/public/certificates/{verificationCode}` and the Web page `/certificates/verify/{code}`), resolved against whatever public hostname serves the app. Operator action needed is limited to ensuring `AllowedHosts`/DNS/routing for the public hostname is correct in production — not a certificate-specific setting.
- [x] Health endpoints are available. `/health`, `/health/live`, `/health/ready` exist on both API and Web, verified repeatedly this cycle including against a restored database.
- [ ] Logging and monitoring are enabled. **Partial, real gap:** only default ASP.NET Core logging is wired up (no Serilog/Application Insights/OpenTelemetry). `docs/operations/production-monitoring.md` explicitly lists alerting (health-check-fail alerts, Slack/Teams webhooks, uptime monitors) and an admin health dashboard as **TODO, not implemented**. Health endpoints exist, but log aggregation and alerting depend entirely on the hosting platform/container runtime the operator chooses — there is no in-app APM.

## Go/No-Go Decision

Current recommendation: `No-Go`.

All engineering-verifiable items are now complete (staging UAT, defect closure, backup/restore, migrations, health checks, secret hygiene, email provider adapter, background delivery worker). Production deployment remains blocked only on items outside engineering's control:
1. Human/organizational sign-offs: release owner approval, rollback owner assignment, and logging/monitoring/alerting setup on whatever hosting platform is chosen (none of this is in-repo).
2. Docker Compose staging re-validation remains deferred per the release owner's own instruction — not currently blocking, but still an open item before this checklist can claim full staging parity.
3. Real SMTP credentials and sending-domain approval (SPF/DKIM, provider account, sender reputation) must be provisioned operationally before `Email__Provider=Smtp` is set in production — the adapter and worker are code-complete and tested, but nobody has configured a live provider account yet.
