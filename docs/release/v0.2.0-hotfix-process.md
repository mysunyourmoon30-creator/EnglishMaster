# v0.2.0 Hotfix Process

## Branch Naming

Use:

```text
hotfix/v0.2.x-short-description
```

Examples:

- `hotfix/v0.2.1-login-failure`
- `hotfix/v0.2.2-public-search-safety`

## Allowed Hotfix Scope

Allowed:

- Production outage.
- Security or data privacy defect.
- Public draft content exposure.
- Quiz answer exposure.
- Data loss or corruption.
- Broken login, admin, learner, health, import, publishing, or reporting workflow with no acceptable workaround.

Not allowed:

- New features.
- Large refactors.
- Cosmetic changes.
- Deferred known limitations unless they become a release-blocking defect.

## Build And Test Requirement

Every hotfix must pass:

```powershell
dotnet restore
dotnet build --no-restore
dotnet test --no-restore
```

Add focused tests for the defect when practical.

## Security Review Requirement

Security or privacy hotfixes require review for:

- Authorization and permission checks.
- Learner-owned data scoping.
- Public content visibility.
- Quiz answer exposure.
- Secrets, tokens, connection strings, and sensitive auth fields.
- Error messages and logs.

## Tag Format

Use patch tags:

- `v0.2.1`
- `v0.2.2`

Do not create or push tags until the hotfix is approved.

## Rollback Condition

Rollback if the hotfix causes outage, data loss, migration failure, private data exposure, public draft exposure, or a critical workflow regression that cannot be immediately corrected.

## Communication Note

Every hotfix should include:

- Incident or defect summary.
- Affected users/workflows.
- Risk assessment.
- Test evidence.
- Deployment window.
- Rollback owner.
- Monitoring owner.
