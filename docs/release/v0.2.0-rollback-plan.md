# v0.2.0 Rollback Plan

## Database Rollback Concept

Take a database backup before applying migrations. Prefer restoring from backup for production rollback unless a down migration has been reviewed and rehearsed.

## File Storage Rollback Concept

Back up media and publishing artifact storage before deployment. Restore both folders when rolling back data that references files.

## App Version Rollback

Keep the previous deployable app package available. Roll back application binaries first if no database migration has been applied.

## Migration Rollback Caution

Do not run destructive migration rollback commands in production without a reviewed plan. Schema rollback can lose data or break newly created records.

## Restore From Backup

1. Stop write traffic or put the app in maintenance mode.
2. Restore the database backup.
3. Restore media and publishing storage if needed.
4. Deploy the previous application version.
5. Reapply environment configuration for the previous version.

## Post-Rollback Smoke Test

- Login.
- Admin dashboard.
- Public home and search.
- Health live and ready endpoints.
- Create/read a simple admin content record.
- Confirm no new v0.2.0-only workflow is required by the rolled-back version.
