# v0.3.0 Rollback Plan

## Database Rollback Concept

Take a database backup before applying v0.3.0 migrations. Prefer restoring from backup for production rollback unless a down migration has been reviewed and rehearsed. This is especially true for v0.3.0: three of the migrations in this release (`20260710001231_AddMediaModule`, `20260710070507_AddPronunciationModule`, `20260712234616_AddQuizModule`) were edited during this cycle to change FK `OnDelete` behavior (`SetNull` → `NoAction`/`ClientSetNull`) to fix SQL Server error 1785 cascade-path failures. Do not attempt an EF Core down-migration through these three without reviewing the current `ON DELETE` semantics first — a down migration could reintroduce the original multi-cascade-path FK definitions and fail identically to how the fresh-database deploy originally failed (see `docs/testing/v0.3.0-defect-log.md`, V030-DEF-001/002/003).

Two more migrations are new in this cycle and are purely additive (`CREATE TABLE`, no destructive changes): `20260717163726_AddLearningActivityLogTable` and `20260717164454_AddProgressMotivationPracticeTables` (16 tables). Rolling these back via down-migration would drop tables backing the Recommendations/Practice/Goals/Study Plan/Motivation/Reports domains — restoring from backup is strongly preferred over a schema down-migration for any rollback that needs to undo these.

## File Storage Rollback Concept

Back up media and publishing artifact storage (`Media:LocalStoragePath` config, or the durable storage path configured for the target environment) before deployment. Restore both folders when rolling back data that references files. No new file-storage-affecting modules were added in v0.3.0 beyond what v0.2.0 already covered (certificates and analytics in this release are DB/API only, not file-backed).

## App Version Rollback

Keep the previous deployable app package (`v0.2.0` or `v0.3.0-rc.1`, depending on rollback target) available. Roll back application binaries first if no database migration has been applied yet.

## Migration Rollback Caution

Do not run destructive migration rollback commands in production without a reviewed plan. Schema rollback can lose data or break newly created records. This applies with extra weight to the three cascade-path-fixed migrations above.

## Restore From Backup

1. Stop write traffic or put the app in maintenance mode.
2. Restore the database backup (see `docs/operations/database-backup-restore.md` for command syntax; see `docs/testing/v0.3.0-backup-restore-verification.md` for a verified restore procedure and evidence from a local dry run against this release's schema).
3. Restore media and publishing storage if needed.
4. Deploy the previous application version.
5. Reapply environment configuration for the previous version.

## Post-Rollback Smoke Test

- Login (SuperAdmin and at least one non-privileged role).
- Admin dashboard.
- Public home and search.
- Health live and ready endpoints.
- Create/read a simple admin content record.
- Confirm no v0.3.0-only workflow is required by the rolled-back version — specifically: certificate template management (`/admin/certificate-templates`), notification delivery admin pages, and the Recommendations/Practice/Goals/Study Plan/Motivation/Reports student-facing areas, all of which are new or corrective in v0.3.0 and depend on the migrations discussed above.
