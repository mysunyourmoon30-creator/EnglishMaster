## Decision

**Approved 2026-07-18.** Release owner approval was given directly by the project owner in chat, closing the last gate on the tag itself. The `v0.3.0` tag is being created and pushed per the commands below. This is a tag/freeze of the release artifact, not a production deployment — actual go-live still follows `docs/release/v0.3.0-production-go-live-checklist.md`'s separate process and its currently-open items (monitoring/alerting, real SMTP provisioning) before this tag is deployed anywhere live.

**Docker Compose staging re-validation: attempted 2026-07-18, abandoned due to a hardware compatibility issue, not pursued further.** BIOS virtualization (SVM Mode) was enabled to unblock Docker, which succeeded. Docker Desktop was installed. But enabling the required Windows "Virtual Machine Platform" feature and rebooting caused the machine to fail to boot (`0xc0000001`, Windows Recovery screen) — resolved by disabling SVM Mode again in BIOS, after which the machine booted normally. This points to a real firmware/driver incompatibility on this specific machine (ASUS TUF Gaming FX505DT) between virtualization and stable boot, not a configuration mistake — SVM Mode likely ships disabled by default on this hardware for exactly this reason. Given the release does not otherwise depend on Docker Compose parity (the local non-Docker path already validated all application-level behavior — permissions, workflows, data isolation, migrations, and all 10 defects found and fixed), risking the development machine's bootability further for this one checkbox is not justified. This gate is considered **closed as "attempted, not achievable on available hardware without further investigation"** rather than pending. Container/networking-specific behavior (Dockerfiles, `docker-compose.staging.yml` health checks, SQL Server Linux container specifics) remains unvalidated and should be picked up on different hardware or in CI if that assurance is ever needed.

V030-DEF-009 is resolved: the certificate template admin CRUD UI is built and verified (see `docs/release/v030-def-009-certificate-admin-ui-scope.md` for scope, and `docs/testing/v0.3.0-defect-log.md` for verification evidence). An admin-wide issued-certificates browser (across all students, not just "my own") remains out of scope for v0.3.0 — deferred as a future enhancement, not a release blocker.

Backup/rollback readiness is resolved for this pass: see `docs/release/v0.3.0-rollback-plan.md` and `docs/testing/v0.3.0-backup-restore-verification.md`. Production-scale backup automation/retention remains separate operational work, not a code-level blocker.
