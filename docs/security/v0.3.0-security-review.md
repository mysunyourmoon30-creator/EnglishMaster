# v0.3.0 Security Review

## Security Risk Summary

The selected scope is acceptable if secret handling, permission enforcement, public verification safety, and learner data ownership are treated as release-blocking requirements.

## Planned Rules

| Area | Rule |
| --- | --- |
| Email admin pages | Require dedicated email/settings permission. |
| Email settings | Never expose provider secrets in API responses or UI state. |
| Email logs | Store bounded sanitized failures; no passwords, tokens, connection strings, or full provider payloads. |
| Notification delivery | Require notification/email delivery permission for admin actions and retries. |
| Admin certificate management | Require dedicated certificate management permission. |
| Student certificate list | Require authentication and scope to the current student. |
| Public certificate verification | Expose only public-safe certificate metadata. |
| Analytics endpoints | Require reporting/analytics permission, except learner-owned summaries scoped to the current student. |

## Email Security Risks

- Provider credentials leaking through settings endpoints, logs, error messages, or UI state.
- Retry endpoints being used for spam-like behavior.
- User-controlled template content causing HTML/script injection.
- Provider-specific errors exposing internal infrastructure details.

Mitigations:

- Store secrets only in environment variables or secret managers.
- Return masked configuration status only.
- Require permissions for send/retry/settings.
- Sanitize or constrain template content.
- Rate-limit or bound manual retry operations.

## Certificate Security Risks

- Forged completion state.
- Guessable verification codes.
- Public verification leaking private learner data.
- Cross-user certificate access.
- Duplicate or unauthorized certificate issuance.

Mitigations:

- Use server-side completion rules.
- Generate high-entropy verification codes.
- Return public-safe verification DTOs.
- Enforce student ownership and admin permissions.
- Enforce uniqueness for issued certificates.

## Analytics Security Risks

- Sensitive learner detail exposed through admin dashboards.
- Cross-user learner analytics exposure.
- Raw metadata or auth-sensitive fields leaking into exports.
- Over-broad reporting permissions.

Mitigations:

- Return aggregates and bounded summaries.
- Scope student analytics to authenticated student profile.
- Exclude raw sensitive metadata from analytics and exports.
- Require analytics/reporting permissions for admin views.

## Security Go / No-Go

`Go with mandatory controls`.

Implementation should not be accepted without authorization tests, public verification safety tests, secret masking tests, and student ownership tests.
