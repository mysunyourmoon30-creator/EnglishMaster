## Known Limitations

1. **LocalDB, not production SQL Server.** This proves the backup/restore *mechanism and app behavior* work correctly against the current v0.3.0 schema, but does not validate production-specific concerns (backup compression, larger dataset timing, network storage paths, retention automation, Azure SQL-specific behavior if applicable). Those remain the responsibility of whoever operates the actual production SQL Server instance.
2. ~~File storage backup not exercised.~~ **Resolved** — see "File Storage Backup/Restore Drill" above. A real file was uploaded, backed up, restored to an isolated location, and verified byte-identical via checksum after being served through a fresh app instance.
3. **No production-scale timing data.** The backup/restore in this drill completed in well under a second because the test dataset is small (both DB and the single test media file). Production backup/restore duration at real data volume is unknown and should be measured separately before relying on this for an RTO (recovery time objective) commitment.
4. **Backup scheduling/automation not covered.** This verifies that a manual backup/restore cycle works; it does not verify or configure any automated backup schedule, off-host storage, or retention policy for production — those are operational setup tasks tracked separately (see `docs/operations/backup-restore-checklist.md` "Daily/Weekly Checks").

## Conclusion

The core backup/restore mechanism is proven to work correctly against the current v0.3.0 database schema **and** file storage: a checksummed database backup can be taken, verified, restored to an isolated database, and the application starts and functions correctly (including the new v0.3.0 certificate admin UI) against the restored data with zero data loss in the spot-checked tables — and a real uploaded file survives the same backup/restore cycle byte-identical, verified by checksum through a live app instance. This closes both the "has restore ever been tested" gap called out in `docs/operations/backup-restore-checklist.md`'s Blockers section and the file-storage-specific gap flagged in the original pass of this document, for this dataset and environment. Production-specific backup automation, scheduling, retention, and RTO timing at real data volume remain separate operational setup work, not blocked on this verification.
