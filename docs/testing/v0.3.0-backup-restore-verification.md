# v0.3.0 Backup/Restore Verification

## Status

`Database Backup/Restore Verified (Local Dry Run)` — file storage backup not exercised (no real media files in this dataset; see Known Limitations).

## Purpose

Executes the "Weekly Checks" and "Before Migration" items from `docs/operations/backup-restore-checklist.md` and the "Restore To Staging First" procedure from `docs/operations/database-backup-restore.md` against the live v0.3.0 schema, to move the `v0.3.0-final-tag-prep.md` "Backup/rollback readiness" gate from `Not Verified` to real evidence. This is the first time this release's backup/restore procedure has actually been executed rather than just documented.

## Known Limitations

1. **LocalDB, not production SQL Server.** This proves the backup/restore *mechanism and app behavior* work correctly against the current v0.3.0 schema, but does not validate production-specific concerns (backup compression, larger dataset timing, network storage paths, retention automation, Azure SQL-specific behavior if applicable). Those remain the responsibility of whoever operates the actual production SQL Server instance.
2. **File storage backup not exercised.** No real media files were uploaded during this v0.3.0 validation cycle (test fixtures were seeded directly via SQL/API, not file upload), so `Media:LocalStoragePath` / publishing artifact backup-restore was not run end-to-end here. The documented procedure in `docs/operations/backup-restore-checklist.md` and `docs/release/v0.3.0-rollback-plan.md` still applies but is unverified with real files in this pass.
3. **No production-scale timing data.** The backup/restore in this drill completed in well under a second because the test dataset is small. Production backup/restore duration at real data volume is unknown and should be measured separately before relying on this for an RTO (recovery time objective) commitment.
4. **Backup scheduling/automation not covered.** This verifies that a manual backup/restore cycle works; it does not verify or configure any automated backup schedule, off-host storage, or retention policy for production — those are operational setup tasks tracked separately (see `docs/operations/backup-restore-checklist.md` "Daily/Weekly Checks").

## Conclusion

The core backup/restore mechanism is proven to work correctly against the current v0.3.0 database schema: a checksummed backup can be taken, verified, restored to an isolated database, and the application starts and functions correctly (including the new v0.3.0 certificate admin UI) against the restored data with zero data loss in the spot-checked tables. This closes the "has restore ever been tested" gap called out in `docs/operations/backup-restore-checklist.md`'s Blockers section, for this dataset and environment. Production-specific backup automation, scheduling, retention, and file-storage backup remain separate operational setup work, not blocked on this verification.
