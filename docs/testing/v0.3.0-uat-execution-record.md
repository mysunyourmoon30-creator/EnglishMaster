Since `DevelopmentSeed__Enabled=false`, the following fixtures were created directly against the local Staging environment to exercise the scenarios below. These are local-only, disposable LocalDB records, not shipped anywhere:

- User `student.qa@englishmaster.local` â€” no roles (plain learner account).
- User `viewer.qa@englishmaster.local` â€” `Viewer` role (12 permissions), later also granted a custom `CertificateReader` role (`certificates.read` only) to test the read/write permission split.
- Custom role `CertificateReader` â€” `certificates.read` only, created to test that permission independent of `certificates.manage` (no seed role isolates the two).
- Course "QA Test Course" (published, active, no lessons â€” sufficient since certificate generation only checks `CourseProgress`, not lesson-level completion).
- `CourseProgress` row for `student.qa` on that course, `Status=Completed`, `ProgressPercent=100` (seeded directly via SQL â€” course completion mechanics/lesson trees are a separate, already-covered area, not what this pass was testing).
- Certificate template `qa-completion` (active).
- One issued certificate for `student.qa` on the QA Test Course.
- Two `EmailMessage` records: one queued-then-processed (ends `Sent`), one seeded directly as `Failed` to test the retry flow.
