# v0.3.0 Staging Validation Results

## Status

`Run - One Open Gap (V030-DEF-009)`

## Summary

Staging validation for `v0.3.0-rc.1` was executed on 2026-07-17/18 against a **local, non-Docker Staging-configured deployment**, not the documented `docker-compose.staging.yml` path. Docker was unavailable on the validation machine (not installed, and a prior attempt to enable virtualization in BIOS/UEFI failed — the firmware settings screen could not be reached), so validation used the fallback documented in `docs/deployment/local-run.md` ("If Docker is unavailable, continue using the normal .NET local workflow"): `dotnet run` for both API and Web with `ASPNETCORE_ENVIRONMENT=Staging`, against a fresh LocalDB database (`Server=(localdb)\mssqllocaldb;Database=EnglishMasterStaging`).

This was the first time in the project's history that the application was run end-to-end against a freshly migrated database. That surfaced **10 real defects** (see `docs/testing/v0.3.0-defect-log.md`, V030-DEF-001 through V030-DEF-010). The 8 Blocker-severity defects (V030-DEF-001–008) were fixed, verified, and are now **committed and pushed** (`5d30fc0` on `origin/codex/v0.3.0`). All 8 are application/migration-code defects that would reproduce identically under the Docker Compose path — none are artifacts of skipping Docker. A follow-up deep UAT pass (see `docs/testing/v0.3.0-uat-execution-record.md`) exercised notification delivery, certificates (via direct API, since no admin UI exists), and analytics permission/scoping in full; it found no further Blockers, confirmed V030-DEF-009 (no certificate admin UI) as a real open gap, and surfaced one Low-severity UX finding (V030-DEF-010, raw HTTP error text on permission-denied pages).

## Decision

Current staging result: `Run - One Open Gap (V030-DEF-009)`.

Eight Blocker-severity defects were found, fixed, verified, and are now committed and pushed (`5d30fc0`). A full deep UAT pass found no further Blocker or High-severity functional defects. Two gaps remain:

1. **V030-DEF-009** (High) — no certificate admin UI exists. Backend logic is fully verified correct via direct API testing. This needs a release-owner decision: build the UI before shipping, or explicitly accept API-only certificate management for v0.3.0.
2. **V030-DEF-010** (Low) — permission-denied pages show raw HTTP status text instead of a friendly message. Cosmetic, not release-blocking.

Final tag and production go-live remain blocked until:
1. The regression suite gap that let 8 Blocker defects (3 migration bugs + a large slice of never-migrated tables + a broken login/auth chain) go undetected is addressed — at minimum, an integration test that migrates a fresh empty database end-to-end and exercises a real login should exist going forward, since none of this was caught by the existing 383 tests.
2. V030-DEF-009 is triaged (build the UI, or explicitly accept API-only certificate testing for this release).
3. This UAT pass is run against a real Docker-based or hosted staging environment at least once before production go-live, since this pass used local LocalDB rather than the documented `docker-compose.staging.yml` path (the Docker container/networking layer itself remains unvalidated).
4. Formal product/engineering/security/operations sign-off per `docs/testing/v0.3.0-uat-plan.md` is completed — this pass produced the evidence but sign-off itself is a human decision.
