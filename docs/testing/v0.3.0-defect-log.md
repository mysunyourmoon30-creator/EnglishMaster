# v0.3.0 Defect Log

## Status Summary

Staging validation was executed on 2026-07-17 against a local (non-Docker) Staging-configured deployment: `dotnet run` with `ASPNETCORE_ENVIRONMENT=Staging` against a fresh LocalDB database (`EnglishMasterStaging`), since Docker was unavailable and Docker Desktop install was blocked by prior BIOS/virtualization issues. See `docs/testing/v0.3.0-staging-validation-results.md` for the full environment note.

This was the **first time the application has ever been run end-to-end against a freshly created database**. That surfaced 10 real defects. The 8 Blocker-severity defects (V030-DEF-001 through V030-DEF-008) are fixed, committed (`5d30fc0`), and pushed to `origin/codex/v0.3.0`. All 8 fixes would have reproduced identically on the Docker-based staging path — none are artifacts of the no-Docker workaround. A subsequent deep UAT pass (notification delivery, certificates via direct API, analytics role matrix and scoping — see `docs/testing/v0.3.0-uat-execution-record.md`) found no further Blocker or High defects, but confirmed V030-DEF-009 (missing certificate admin UI) is a real, unfixed gap, and surfaced one new Low-severity UX finding (V030-DEF-010). Both are now fixed: V030-DEF-010 with a one-file friendly-error-message fix, and V030-DEF-009 by building the certificate template admin CRUD UI per the scope doc's recommendation. As of this follow-up pass, all 10 defects found during v0.3.0 staging validation are closed.
