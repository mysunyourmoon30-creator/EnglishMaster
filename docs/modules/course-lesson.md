# Course Lesson

## Purpose

`CourseLesson` is the ordered join entity between `Course` and `Lesson`. It lets a course contain many lessons while allowing the same lesson to appear in many courses. It is not an independent business module and has no standalone Blazor route; it is managed from the Course detail page.

## CourseLesson Fields

| Field | Description |
| --- | --- |
| `Id` | Unique relation identifier. Generated by the domain and configured with `ValueGeneratedNever` in EF Core. |
| `CourseId` | Required parent course identifier. |
| `LessonId` | Required lesson identifier. |
| `LessonTitle` | Lesson title included in response DTOs. |
| `LessonSlug` | Lesson slug included in response DTOs. |
| `LessonSummary` | Lesson summary included in response DTOs. |
| `LessonCefrLevel` | Optional lesson CEFR level included in response DTOs. |
| `SortOrder` | Display order inside the course. Must be zero or greater. |
| `IsRequired` | Indicates whether the lesson is required in the course path. |
| `CreatedAt` | Relation creation timestamp. |
| `UpdatedAt` | Relation last update timestamp. |

## Course Relationship

`CourseLesson` belongs to one `Course`.

- `Course` owns the `Lessons` collection.
- EF Core maps `Course` to many `CourseLesson` records with cascade delete from course to relation rows.
- The domain method `Course.AddLesson` creates relation rows and prevents duplicates.
- `Course.RemoveLesson` removes an existing relation and updates the course timestamp.

## Lesson Relationship

`CourseLesson` references one `Lesson`.

- The Application layer validates that the lesson exists and is active before it can be added.
- The database relationship uses restrict delete behavior for `LessonId`, so referenced lessons are protected.
- Response DTOs enrich the relation with lesson title, slug, summary, and CEFR level.

## Domain Rules

- `Id`, `CourseId`, and `LessonId` cannot be empty GUIDs.
- `SortOrder` must be zero or greater.
- `IsRequired` can be changed by adding an existing lesson relation again.
- `Reorder` changes `SortOrder` and updates `UpdatedAt`.
- Duplicate `LessonId` values inside one course are prevented by both domain behavior and a unique index on `(CourseId, LessonId)`.
- Domain code does not reference EF Core, ASP.NET Core, Blazor, or infrastructure.

## API Endpoints

| Method | Route | Purpose |
| --- | --- | --- |
| `GET` | `/api/v1/courses/{courseId}/lessons` | List lessons in a course ordered by `SortOrder`. |
| `POST` | `/api/v1/courses/{courseId}/lessons/{lessonId}` | Add a lesson to a course. Optional query parameters: `sortOrder`, `isRequired`. |
| `DELETE` | `/api/v1/courses/{courseId}/lessons/{lessonId}` | Remove a lesson from a course. |
| `POST` | `/api/v1/courses/{courseId}/lessons/reorder` | Reorder all lessons in a course. |

Full status codes and examples are documented in [Course API](../api/course-api.md).

## Blazor UI

Course lessons are managed inline on `/admin/courses/{id:guid}`.

- Active lessons are loaded into a selector.
- Already-related lessons are hidden from the selector.
- Admins can mark a lesson required or optional when adding it.
- Admins can remove lessons.
- Admins can reorder lessons using Up and Down actions.
- Loading, empty, error, and validation states are shown through the Course detail page.

## Test Coverage

Current tests cover:

- Creating course lesson relations through `Course.AddLesson`
- Duplicate lesson prevention inside one course
- Updating `IsRequired` when the same lesson is added again
- Reordering `CourseLesson` rows
- Listing, adding, removing, and reordering course lessons through the API integration test

## Known Limitations

- There is no standalone `/admin/course-lessons` page.
- There is no standalone `GET /api/v1/course-lessons/{id}` route.
- Reorder requires the submitted id list to exactly match the course's current lesson relation set.
- Relation rows are removed from the course instead of soft-deactivated.
